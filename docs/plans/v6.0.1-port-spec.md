## 3. session-start: add git-based update path WITHOUT removing the curl path

**File:** `hooks/session-start`

The v6.0.1 version **replaced** the curl-based update check with a git-based one. That was a regression for Claude Code and Cursor marketplace users ‚Äî their plugin install is not a git repo, so the git check exits silently and they receive no update notification.

The correct implementation has **both paths**:

1. **Git path** (for Codex and self-hosted clones): if `PLUGIN_ROOT` is a git repo, run `git fetch origin` + `git merge --ff-only origin/main`. Show "What's New" if VERSION changed.
2. **Curl path** (for Claude Code / Cursor marketplace installs): if `PLUGIN_ROOT` is NOT a git repo, run the existing curl check against the remote `plugin.json` and tell the user to run `/plugin update`.

The v6.0.1 session-start has all the right building blocks ‚Äî add back the curl fallback from v6.0.0 as the `else` branch. Specifically:

From v6.0.0, the curl branch (lines 50‚Äì83) checks:
- local version from `installed_plugins.json` or `plugin.json`
- remote version from `curl -sf --max-time 3 https://raw.githubusercontent.com/.../plugin.json`
- semver comparison
- injects update notice telling user to run `/plugin update superpowers-optimized`

Suggested `check_for_updates()` structure after the port:

```
check_for_updates() {
    # shared: cache check (24h TTL) ‚Äî skip everything if cache is fresh
    # branch A: if PLUGIN_ROOT is a git repo
    #   ‚Üí git fetch + ff-only merge + VERSION comparison + What's New
    # branch B: else (marketplace install ‚Äî no git repo)
    #   ‚Üí curl remote plugin.json + semver compare + notify user to /plugin update
    # shared: write cache file on any check
}
```

Also port from v6.0.1:
- `is_auto_update_enabled()` function (reads `SUPERPOWERS_AUTO_UPDATE` env var + `~/.config/superpowers/update.conf`)
- old_version / new_version capture + `readWhatsNew` awk block
- The `update_notice` variable injection into `session_context`

## 5. hooks/codex-hooks.json (new file ‚Äî port with fixes)

**File:** `hooks/codex-hooks.json` (new untracked file in v6.0.1)

Port this file, but **rewrite the command** before committing. The v6.0.1 version has a single-line shell command crammed into a JSON string with nested escaping and `ls` output parsing. Specific problems:

- Parses output of `ls -d "$HOME/.codex/plugins/cache"/*/superpowers-optimized/*` ‚Äî this path looks like it may be Claude Code's internal plugin cache layout, not Codex's. Verify whether Codex actually stores plugins at `~/.codex/plugins/cache` before keeping this path.
- If that path is wrong/unused, the first branch of the if/elif chain always fails and adds no value ‚Äî it's dead code.
- Calling `run-hook.cmd` (the CMD/bash polyglot wrapper) on Unix/macOS is unnecessary indirection. On the only platform where the polyglot wrapper matters (Windows), Codex lifecycle hooks don't run anyway.

Proposed cleaner command for the JSON:
```
if [ -f \"$HOME/.codex/superpowers-optimized/hooks/session-start\" ]; then \"$HOME/.codex/superpowers-optimized/hooks/session-start\"; elif [ -f \"$HOME/.codex/superpowers/hooks/session-start\" ]; then \"$HOME/.codex/superpowers/hooks/session-start\"; fi
```

This: calls `session-start` directly (no polyglot wrapper needed on Unix), handles both the new and old install paths, drops the unverified plugins/cache path.

The matcher `"startup|resume|clear|compact"` should also be reconsidered. Firing the hook on every `clear` and `compact` operation is wasteful ‚Äî the 24h cache prevents actual git fetches, but it still executes shell commands on context operations. Consider whether `"startup|resume"` is sufficient.
